## Implementation notes

- **ingext-health-monitor (1.0.0 → 1.0.1)** — no behavioural change. Every check's
  query was extracted out of the prose steps into a dedicated file under
  `assets/queries/` (one file per check: `login_activity.txt`, `data_ingress.promql`,
  `router_pipe_errors_drops.promql`, `egress.promql`, `ingestion_anomaly.promql`,
  `queue_backlog.promql`). `SKILL.md` gained a **Query assets** table mapping each
  check to its file, plus a `→ assets/queries/...` pointer on each step; the inline
  queries remain as the readable mirror. Check logic, the overall-health rules, the
  HTML report structure, and styling are untouched.

- **ingext-promql (1.0.0 → 1.0.2)** — expanded the time-format guidance to match verified
  behaviour on the live instance: `time` / `from` / `to` accept **all three** of relative offsets
  (`-24h`, `-1617600s`, `-0h`), **epoch milliseconds** (`1781827200000`), and **RFC3339/ISO-8601**
  (`2026-06-19T00:00:00Z`) — all three return identical results for the same instant (verified by
  running one query three ways). For an exact boundary (e.g. a billing-month start) an absolute
  epoch-ms/RFC3339 timestamp is now recommended since it pins the instant with no drift; relative
  offsets still work. (An earlier same-day observation that absolute timestamps were rejected was
  invalidated when the metrics backend was updated mid-session; the parser now accepts them.) No
  metric/catalog changes.

- **billing-csv (new, 1.0.0)** — generates a per-client billing CSV by querying the
  datalake/resource tables and platform metrics directly, with every query kept as a
  bundled asset:
  - **Bundled `assets/Microsoft365_LicenseTable.csv`** — paid M365 SKU reference
    (`Product name, SKUID, String ID`); all free/trial/developer SKUs removed, so every
    row is a paid license.
  - **Billable-user counting** — `assets/queries/office365_licensed_users.kql` returns
    one row per `(Member user, skuId)` from the `office365User` snapshot table; the paid
    filter (match `skuId` against the CSV) and distinct-user count are done downstream
    (the engine drops dynamic-array checks computed after a `make_set`, so the match must
    happen outside KQL). `assets/queries/googleworkspace_users.kql` is a placeholder —
    no Google Workspace table exists in the tenant yet.
  - **Data-volume metrics (PromQL)** — `assets/queries/egress_bytes_by_dest.promql`
    (datalake vs eventwatch stored bytes) and `assets/queries/data_dropped_bytes.promql`
    (dropped bytes, scoped to `router|datasink` to avoid the pipe/router double-count).
  - **Billing-window handling** — the metrics tools accept only relative offsets (epoch
    ms / RFC3339 are rejected), so a billing month is expressed as a day-count window
    plus an evaluation instant of `-<offsetSeconds>s` (now − cycle end); a helper and a
    partial-month path (evaluate at `-0h`) are documented in `SKILL.md`.
  - Verified end-to-end against the live tenant with May 2026 and June 2026
    (month-to-date) dry runs.
