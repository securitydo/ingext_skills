# 2026-06-22 — office-user-investigation geo accuracy fix; azure-user-signin-investigation goes report-free

Two skill updates. `office-user-investigation` now scopes its per-IP analysis to real login
events and plots the map from the platform's live GeoIP enrichment instead of the stale
offline database. `azure-user-signin-investigation` no longer depends on three saved FPL
reports — it runs the equivalent KQL directly against the Azure tables, so the only
dependency is the `AzureSigninLogs` / `AzureAuditLogs` datalake tables. For what each skill
does and its trigger phrases, see its section in the root `README.md`.

## Implementation notes

- **ingext-health-monitor (1.0.1)** — description-only update (no version change, no behaviour
  change). The `description` frontmatter was reformatted from a single long line into a folded
  block scalar and lightly tightened for readability; the trigger phrasing is unchanged in
  substance. The `cowork/ingext-health-monitor.skill` package was rebuilt to match. (This also
  reconciled a pre-existing source/package drift for this skill.)

- **office-user-investigation (1.0.0 → 1.0.1)** — source-IP accuracy fix plus
  Step 1 / default-window adjustments. No layout/heuristic changes.
  - **`ip_summary` query is now login-only.** Added `where Operation in ("UserLoggedIn","UserLoginFailed")`
    before the per-IP summarize, so service-side operations (OneDrive/SharePoint `FileAccessed`,
    etc.) accessed from internal Microsoft datacenter IP ranges are no longer counted as user
    sign-in locations. Removed the `sends=countif(Operation=="Send")` aggregate (a send can't
    occur in a login-only set).
  - **Per-IP geo + ISP carried from the platform.** The query now groups by
    `ClientIP, Country, CC, City, ISP, Lat, Lon` (from the `_ip` enrichment) instead of
    `ClientIP` alone, tying each address to a single country/city/ISP.
  - **Map uses platform coordinates; offline GeoLite2 is a fallback.** `build_report.py`
    plots each IP from the query-supplied `CC`/`City`/`Lat`/`Lon` when present and only calls
    the bundled offline GeoLite2 DB for IPs the query didn't geolocate. This removes the
    stale-database discrepancy (the GeoLite2 PyPI snapshot is ~2018 and unmaintained) — e.g.
    a Microsoft-owned IP that GeoLite2 placed in Belgium is correctly a London/UK datacenter
    address, and such service IPs no longer appear at all once login-only filtering is applied.
  - **"Top source IPs" table** gained **Country**, **City** and **ISP** columns.
  - **Map caveat text** updated to state coordinates come from live GeoIP enrichment with an
    offline fallback (still flagged approximate — verify with live threat intel before
    attribution).
  - **Added `assets/countries/BEL.geo.json`** (Belgium outline) so the map draws a border if
    an IP ever resolves there instead of emitting a missing-outline warning.
  - Backward compatible: if a tenant returns the old `ip_summary` shape (no `CC`/`Lat`/`Lon`),
    the build falls back to offline GeoLite2 exactly as in 1.0.0. No changes to the `workload`,
    `operation`, `inbox_rules`, `oauth` or `timeline` queries, the report layout, or the
    BEC/verdict heuristics.
  - **Verified** by rebuilding the kun@fluencysecurity.com investigation (19–22 Jun 2026): the
    three Microsoft datacenter IPs dropped out (no login events), the two genuine Verizon IPs
    plot to Towson, US and Ashburn, US, and the verdict moved from a false "suspicious / foreign
    access" to low / no foreign access.
  - **Step 1 checks the table, not the index.** Step 1 now calls `list_data_tables` and confirms
    the `Office365` table appears in `streamTables`, replacing the previous `list_indexes` /
    `datalakeIndex "Office365"` check (and the `describe_schema` mention). Pipeline diagram,
    Step 1 body, and the failure-modes row updated to match; the trailing description reference
    to "an Office365 index" became "an Office365 table".
  - **Default 7-day window.** If no time range is supplied, the skill defaults to the last
    7 days (`from = now - 604_800_000`, `to = now`) and proceeds without asking; `username`
    remains required. Other presets (24h/7d/30d/90d) still offered on request.

- **azure-user-signin-investigation (1.0.0 → 1.0.1)** — removed the dependency on three saved
  FPL reports; the skill now runs the equivalent KQL directly. Behaviourally the report output
  is unchanged.
  - **Extracted the three report queries into `assets/queries/`** — `signin.kql`,
    `dir_changes_initiated.kql`, `dir_changes_targeting.kql` (the trimmed projections the
    reports compiled to). Each uses `{USER}` / `{FROM}` / `{TO}` placeholders and carries its
    own `TimeGenerated between (...)` bounds.
  - **Pipeline rewired from `run_report` to `kql_search`.** `list_reports` / `run_report` /
    `get_report_task` / `get_report_result` are gone; the skill confirms `AzureSigninLogs` and
    `AzureAuditLogs` via `list_data_tables`, substitutes the placeholders, and runs the three
    queries directly. The only dependency is now those two Azure tables.
  - **`build_report.py` learned the `kql_search` shape.** `load_report()` now normalises the
    `kql_search` `{"data":{"Tables":[{"Columns":[{"ColumnName":..}],"Rows":[[..]]}]}}` envelope
    into the same `{"objects":[{"table":{"columns","rows"}}]}` structure the renderer already
    used — so it accepts both the new direct-KQL results and the legacy report JSON (backward
    compatible).
  - **SKILL.md** updated: "Required reports" → "Required data tables", a "Bundled queries"
    table, the new pipeline/steps, failure modes, and layout. Description and triggers updated
    to state the only dependency is the Azure tables.
  - **Verified**: the three extracted queries validate via `validate_kql`, and a build run with
    `kql_search`-shaped inputs renders correctly (KPIs, sign-in table, timeline, profile card).
  - **Report redesign (hybrid style + insights).** `report_template.html` upgraded with
    office-report-style components on the lighter Fluency palette: a **Key Insights** strip of
    accent cards (countries, auth-success rate, busiest source, top application, an
    Entra-risk/Conditional-Access flag, and directory-change count) and a new **Source IPs &
    Geolocation** section with per-IP (events bar, OK/fail, first/last seen, location) and
    per-country breakdown tables. Header/footer corrected from the old "O365 User Investigation"
    label to "Azure AD Sign-in Investigation". `build_report.py` gained `build_geo_breakdown()`
    and `build_insights()`; the risk card surfaces non-empty `RiskLevelDuringSignIn` /
    `RiskState` and `notApplied` Conditional Access. Verified with a multi-IP / multi-country
    sample (a risky foreign sign-in correctly surfaces in the insights and geo tables).
  - **Diagnostic layer (verdict + findings).** Added `build_diagnostics()` — the report now
    *interprets* the same sign-in + directory data instead of only listing it. A **verdict
    banner** (Critical / High / Medium / Low) sits up top, and a **Diagnostic Findings** section
    lists each triggered signal with evidence and a severity pill. Heuristics, all from data
    already collected: Entra risk (`RiskLevelDuringSignIn` / `RiskState`), directory changes
    targeting the account, sign-ins from multiple countries (impossible-travel candidate), failed
    sign-ins (with top error code), legacy authentication clients, Conditional-Access gaps
    (Low — hardening note, not over-weighted), and user-initiated directory changes; privileged
    role context (from `get_azure_user_record`) is noted in the verdict. A risky sign-in combined
    with failures / foreign access / account-targeting changes escalates the verdict to Critical.
    Verified both paths: a suspicious sample → "Critical — Likely compromise indicators"; a clean
    single-country, all-success sample → "Low — Minor findings".
